0일차 - C++

/*
1. C언어: 메모리 추적
보안의 핵심은 메모리다. 변수가 선언될 때 메모리에서 어떤 일이 벌어지는지 눈으로 확인해라.

과제: 정수형(int), 실수형(double), 문자형(char) 변수를 각각 선언하고, 각 변수의 크기(byte)와 메모리 주소를 출력하는 코드를 작성해라.

핵심: sizeof 연산자와 주소 연산자(&)를 사용해라.

질문: int와 double의 크기가 왜 다른지, 그리고 주소값이 왜 일정한 간격이 아닌지 고민해보고 답해라.
-> 숫자를 다루는 자료형. 정수 ( 소수점 없음 )와 실수 ( 소수점 있음 )라는 핵심적 차이 존재
-> 표현하려는 데이터의 성격 ( 정수 vs 실수 )과 정밀도 때문

-> 주소값이 실행되거나 위치에 따라 일정하지 않은 이유는 운영체제 ( OS )의 메모리 관리 방식과 데이터가 저장되는 영역의 특성
1. 운영체제의 동적 메모리 할당 : 프로그램이 실행될 때마다 스택, 힙, 라이브러리 등의 메모리 주소를 임의로 할당하기 때문에 주소값이 일정하지 않음
2. 스택 ( Stack ) 메모리 사용 : 지역 변수는 스택 영역에 저장되며, 함수가 호출될 때마다 할당되고 해제되므로 주소값이 일정하지 않음
3. 힙 ( Heap ) 메모리의 동적 할당 : new 나 malloc 을 사용해 힙 영역에 메모리를 할당한 경우, 운영채재는 현재 사용가능한 가장 적절한 위치에 메모리를 할당
4. 컴파일러의 최적화 : 컴파일러는 속도 향상을 위해 데이터를 레지스터에 보관하거나, 메모리 배치 순서를 최적화할 수 있음. 이로 인해 주소값이 일정하지 않을 수 있음
*/

/*
Sizeof 연산자
컴파일 타임에 크기를 계산하는 연산자. 변수나 데이터 타입의 크기를 바이트 단위로 반환한다.
특징 :
1. 바이트 단위 : 반환값은 1바이트를 기준으로 한다.
2. 컴파일 타임 : 프로그램 실행 전에 크기가 결정된다.
3. 배열 : 배열 이름에 사용하면 배열 전체의 크기 ( 요소의 개수 X 자료형 크기 )를 반환한다.
4. 포인터 : 포인터 변수에 사용하면 포인터 자체의 크기 ( 32비트 : 4바이트, 64비트 : 8바이트 )를 반환한다.
*/

/*
& 연산자
변수나 객체의 메모리 주소를 반환하는 연산자. 주소 연산자라고도 불린다.
특징 :
1. 주소 반환 : 변수 앞에 &를 붙이면 그 변수의 메모리 주소를 반환한다.
2. 포인터와 함께 사용 : 포인터 변수에 &를 사용하여 변수의 주소를 얻고, 포인터를 통해 해당 변수에 접근할 수 있다.
3. 참조 변수 : C++에서는 &를 사용하여 참조 변수를 선언할 수 있다. 참조 변수는 다른 변수의 별칭으로, 해당 변수와 동일한 메모리 주소를 공유한다.
4. 주소값 : & 연산자를 사용하여 얻은 주소값은 메모리 상의 위치를 나타내며, 일반적으로 16진수로 표현된다.
*/

#include <iostream>  // 입출력 라이브러리, input/output stream, c++ 표준 라이브러리
using namespace std; // std 네임스페이스 사용, 표준 라이브러리에 저장된 함수와 객체를 사용할 때 std:: 생략 가능

int main()
{
    int a;
    double b;
    char c;

    cout << "Size of int :" << sizeof(a) << "\nByte:" << &a << endl;
    cout << "Size of double : " << sizeof(b) << "\nByte" << &b << endl;
    cout << "Size of char : " << sizeof(c) << "\nByte" << &c << endl;
    return 0;
}

----------

[잘한 점]

#include <iostream>과 using namespace std;에 주석을 달아 자신이 쓰는 라이브러리의 의미를 정확히 파악하려 한 태도는 아주 훌륭하다.

요구했던 sizeof 연산자와 주소 연산자 &를 문법에 맞게 잘 배치했다.

[부족한 점 및 개선 방향]
하지만, 실무적인 관점에서 자네의 코드에는 수정해야 할 부분과 C++의 중요한 함정이 하나 숨어있다. 코드를 직접 컴파일하고 실행해 봤는지 묻고 싶군.

출력문의 논리적 오류 (오타)
출력문을 보면 "\nByte:" << &a 라고 작성했다. sizeof의 결과가 바이트(Byte) 크기이고, &a의 결과는 메모리가 위치한 16진수 주소(Address)다. 따라서 텍스트를 "\nAddress: "로 수정하는 것이 문맥상 맞다.

메모리 주소 출력의 함정 (char의 특성)
자네의 코드를 그대로 실행하면 int와 double의 주소는 잘 나오겠지만, char c의 주소를 출력하는 cout << &c 부분에서는 이상한 문자가 깨져서 출력되거나 쓰레기값이 나올 거다.
[왜 그런가?]
C++의 cout은 문자형 포인터(char*, 즉 문자의 주소)를 전달받으면, 그것을 단순한 메모리 주소가 아니라 '문자열(String)'의 시작으로 인식해 버린다. 그래서 문자의 끝을 알리는 널 문자(\0)를 만날 때까지 메모리를 무작정 읽어서 출력해버리지.

[더 나은 코드 방식]
이 문제를 해결하려면 컴파일러에게 "이건 문자열이 아니라, 그냥 메모리 주소 공간 그 자체야"라고 명시적으로 알려줘야 한다. 이를 형변환(Casting)이라고 부른다. void* (어떤 타입인지 정해지지 않은 주소)를 사용하면 된다.

// 개선된 코드의 일부
cout << "Size of char : " << sizeof(c) << " bytes\nAddress: " << (void*)&c << endl;

----------

[평가 및 피드백]

첫 번째 질문, int와 double의 크기 차이에 대한 대답은 정답이다. 데이터의 성격과 정밀도에 따라 필요한 공간이 다르다는 핵심을 정확히 짚어냈다. 실수를 표현하려면 정수부, 소수부, 그리고 지수부까지 저장해야 하니 당연히 더 큰 바이트가 필요하지.

두 번째 질문, 주소값이 일정하지 않은 이유에 대한 조사도 아주 훌륭하다. 특히 운영체제의 메모리 관리 방식과 스택(Stack) 영역의 특성을 언급한 점은 칭찬할 만하다. 자네가 언급한 '실행할 때마다 주소가 바뀌는 현상'은 보안 업계에서 ASLR (Address Space Layout Randomization, 주소 공간 배치 무작위화)이라고 부르는 핵심 방어 기법이다. 해커가 메모리 주소를 예측해 공격(Buffer Overflow 등)하는 것을 막기 위해 OS가 일부러 주소를 섞어버리는 거지.

[부족한 점과 핵심 개념 교정]

하지만 자네의 대답에서 실무적인 오점과, 내가 이 질문을 던진 '진짜 의도'가 누락되어 있다.

상황에 맞지 않는 개념: 자네가 짠 코드는 지역 변수만 사용했기 때문에 3번에서 언급한 힙(Heap) 영역의 동적 할당은 이번 결과와 전혀 무관하다. 자신이 작성한 코드가 메모리의 어느 영역을 쓰는지(스택) 명확히 인지해야 한다.

메모리 정렬 (Memory Alignment)과 패딩 (Padding): 변수 a, b, c를 연달아 선언했는데 주소 간격이 변수 크기만큼 딱딱 떨어지지 않는 가장 큰 이유는 컴파일러의 메모리 정렬 때문이다.

CPU는 메모리를 1바이트씩 쪼개서 읽지 않고, 처리 속도를 높이기 위해 4바이트나 8바이트 단위(Word)로 한 번에 읽어 들인다.

그래서 컴파일러는 CPU가 가장 빠르게 데이터를 읽을 수 있는 주소 위치(주로 4나 8의 배수)에 변수를 강제로 끼워 맞춘다.

이때 아귀가 맞지 않아 남는 빈 공간을 패딩(Padding)이라고 부른다. 속도를 위해 메모리 공간 일부를 희생하는 거지.

C/C++로 보안을 하려면, 혹은 해킹 방어 기법을 분석하려면 이렇게 메모리 상에 보이지 않는 '빈 공간'이 존재한다는 사실을 뼛속까지 이해하고 있어야 한다.
